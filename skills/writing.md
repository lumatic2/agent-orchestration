자연스러운 한국어 글쓰기 세션을 시작한다.

$ARGUMENTS 형식: [장르] [주제/내용] [옵션]

옵션: --outline, --draft, --polish, --vault, --brief, --style, --series, --verify

## 0. 인수 처리

### 인수 없이 호출 시 — 인터랙티브 모드

$ARGUMENTS가 비어 있으면 아래 형식으로 시작한다:

```
🔗 다른 크리에이티브 작업에서 이어왔나요?
🎵 음악 → Suno 프롬프트·가사·분위기 붙여넣기
🎨 그림 → 이미지 설명·프롬프트 붙여넣기
(없으면 Enter로 건너뛰기)
```

**연계 컨텍스트가 있으면:**
- 음악에서 이어온 경우: 가사·분위기·서사 추출 → 에세이/시/단편 장르 추천 + 소재·톤 반영
- 그림에서 이어온 경우: 이미지 묘사·감정·배경 추출 → 장면 묘사 또는 브랜드 카피 추천
- 컨텍스트 없으면: 아래 메뉴로 진행

```
어떤 글을 쓰려고 하세요?

✏️ 에세이 — 개인적 경험·생각, 브런치/블로그
📖 소설 — 단편·장편, 묘사·심리 중심
📄 학술 — 논문·보고서, 리뷰, 분석, 객관적 격식체
📢 카피 — 광고·마케팅·랜딩페이지, 심리 트리거·간결성
🔧 기술문서 — 튜토리얼·가이드·README, 명확한 구조
📝 자유 — 장르 미지정, anti-AI 룰셋만 적용

문체 스타일이 있으면 알려주세요.
🖊️ luma(기본) / 🔥 이외수 / 🌿 한강 / 🎵 하루키 / 🧙 롤링 / 📐 유시민 / 🚫 없음

출력 형식도 함께 말해주시면 바로 준비할게요.
📄 마크다운(기본) / 🍵 브런치 / 📝 Word / 🖨️ PDF

플랫폼/매체를 알려주시면 최적화할게요. (선택)
📱 브런치 / 📧 뉴스레터 / 📲 SNS(인스타/X/스레드) / 📄 일반(기본)
```

이후 사용자 답변을 들으며 아래 항목들을 대화 흐름 속에서 자연스럽게 파악한다.
한 번에 2~3개씩 묶어서 묻는다. 면접처럼 하나씩 캐묻지 않는다.
이미 답변에서 유추 가능한 항목은 다시 묻지 않는다.

**파악할 항목:**
- 주제/소재: 무엇에 대해 쓸 건지
- 장르: 에세이 / 소설 / 학술 / 카피 / 기술문서 / 자유
- 플랫폼: 브런치 / 뉴스레터 / SNS / 일반
- 용도/매체: 브런치, 블로그, 과제, 발표, 개인 정리 등
- 독자: 누가 읽을 글인지
- 톤: 가볍게 / 진지하게 / 유머 / 담백하게
- 분량: 짧게(~1000자) / 보통(~2000자) / 길게(3000자+) / 시리즈
- 문체: luma(기본) / 이외수 / 한강 / 하루키 / 롤링 / 유시민 / 없음
- 출력 형식: 마크다운(기본) / 브런치 / Word / PDF
- 경험 유무: 직접 경험인지, 관찰/상상인지 (에세이·소설 한정)

모든 항목이 파악되면 outline 제시로 넘어간다.

### 인수 있을 때 — 빠른 진입

$ARGUMENTS에 장르와 주제가 있으면 대화 단계를 짧게 가져간다.
유추 불가능한 항목만 1~2개 확인 후 outline으로 넘어간다.

--draft, --outline, --polish 옵션이 명시된 경우 대화 단계를 건너뛰고 바로 실행한다.

## 1. 장르 자동 라우팅

대화에서 파악한 내용으로 장르를 결정한다:
- 에세이/브런치/블로그/일상/생각/경험/칼럼 → essay
- 소설/단편/장편/캐릭터/플롯/묘사/시나리오 → fiction
- 논문/보고서/분석/학술/연구/리뷰 → academic
- 카피/광고/랜딩/CTA/마케팅/슬로건/배너 → copy
- 기술문서/튜토리얼/가이드/API/README/문서화 → technical
- 그 외 → free

장르 확정 후 한 줄로 표기:
```
[{장르명}] — {장르 특성 한 줄}
```

장르 결정 후 플랫폼도 결정한다:
- 브런치/브런치스토리/brunch 언급 → platform: brunch
- 뉴스레터/newsletter/구독/메일/레터 언급 → platform: newsletter
- 인스타/인스타그램/X/트위터/스레드/SNS/숏폼 언급 → platform: sns
- 그 외 → platform: general

플랫폼 확정 후 한 줄로 표기:
```
[{플랫폼명}] — {플랫폼 특성 한 줄}
```

## 2. 스타일 가이드 로드

### 2-1. 장르 가이드
`mcp__obsidian-vault__read_note` → `10-knowledge/writing/{장르}.md`
- 성공: 해당 가이드를 컨텍스트로 적용
- 실패: 아래 내장 기본 규칙으로 진행 (별도 안내 불필요)

### 2-2. 문체 가이드 (선택된 경우)
`mcp__obsidian-vault__read_note` → `10-knowledge/writing/styles/{작가명}.md`
- "없음" 선택 또는 미지정 시 → luma 문체를 기본 적용
- 장르 가이드 = 구조·형식 규칙, 문체 가이드 = 문장·어휘·톤 규칙
- 충돌 시 문체 가이드가 우선 (예: 장르는 격식체인데 문체가 구어체면 → 문체 따름)
- `--style {작가명}` 옵션으로도 지정 가능

### 장르별 내장 기본 규칙

**essay**: 1인칭 허용. 구어체적 요소, 고백적·친화적 톤. 경험→성찰→통찰 흐름. 문장 길이 변주로 리듬감.

**fiction**: show don't tell. 감각 묘사 우선. 대화체는 인물 성격 반영. 심리 직접 서술 최소화. 시점 일관성 유지.

**academic**: 격식체(-다/-ㄴ다). 객관적·건조한 톤. 논거→증거→해석 구조. 전문 용어는 첫 등장 시 정의.

**copy**: 첫 문장에서 잡기(3초 룰). 심리 트리거 활용(긴급성/사회적 증거/희소성). 간결·직관·행동 유도.

**technical**: 짧은 문장. 목록 적극 활용. 주어 생략 금지. 코드블록은 맥락과 함께 제시.

**free**: 장르 규칙 없음. anti-AI 룰셋만 적용.

## 2-A. 플랫폼 포맷 가이드

- platform=general이면 이 섹션을 생략한다.

**brunch 포맷:**
- 분량: 1,500~3,000자 권장 (목표 미설정 시 2,000자 기본)
- 제목: 감성적 명사형 or 질문형, 14자 이하
- 소제목: 없거나 최소화 (H2/H3 남발 금지)
- 문단 구성: 3~5문장, 문단 사이 공백 1줄
- 첫 문장: 장면·질문·선언으로 바로 끌어당기기
- 마지막 문단: 열린 여운 (교훈적 결론 금지)
- 완성 후: 커버 이미지 캡션 제안 1줄 추가

**newsletter 포맷:**
- 분량: 800~1,500자 (스캐닝 가능한 길이)
- 구조: 제목 → 도입(1~2문단) → 본문(2~4섹션) → 마무리 → CTA 1개
- 소제목: 각 섹션 명확히 구분 (굵은 글씨 or ##)
- 톤: 구독자와 1:1 대화 — "당신"보다 직접 말걸기 ("지난주에 저는...")
- CTA: 마지막 문단에 행동 유도 1개 ("다음에는 ~에 대해 쓸게요" 또는 링크)
- 분량 기본값: 목표 미설정 시 1,000자

**sns 포맷:**
- 인스타그램: 150자 이내 캡션 + 해시태그 5~10개 별도 제안
- X(트위터): 280자 이내 단일 트윗. 또는 스레드(2~5개, 각 280자)
- 스레드(Threads): 500자 이내, 대화형 톤
- 공통 규칙: 첫 줄이 전부 (더보기 전에 잡아야 함), 이모지 1~2개 허용
- SNS 선택 시 구체적 플랫폼 한 번 확인 후 진행

## 3. Anti-AI 룰셋 (모든 장르 공통, 항상 적용)

### 절대 금지 패턴
- 쉼표 남용: 한 문장에 쉼표 2개 이상이면 문장을 나눠라
- 번역투: ~에 대한, ~되어집니다, ~것이 중요합니다, ~를 가지다, ~를 통한, ~에 있어서
- 어미 반복: 연속 3문장 같은 종결어미 금지 (~다/~다/~다, ~요/~요/~요)
- 접속사 과용: 또한/하지만/그러나/따라서/그리고 → 문장 구조 자체로 연결해라
- 감정 직접 서술: "매우 슬펐다" "정말 감동적이었다" → 상황·행동·감각으로 보여줘라
- 상투적 수식어: 다양한/중요한/필수적인/혁신적인/획기적인/효과적인
- 대명사 오용: 그들/그녀/그것 → 한국어는 주어 생략이 자연스럽다
- 상투적 서론: "오늘날~시대에" / "~의 중요성이 대두되고" / "최근 들어"
- 상투적 결론: "요약하자면" / "결론적으로" / "~해보는 건 어떨까요"
- 나열 남용: "첫째, 둘째, 셋째" 패턴 (3개 이상 기계적 나열 금지)
- 과잉 강조: "정말" "매우" "굉장히" 연속 사용

### 필수 적용 원칙
- 주어-서술어 가깝게 배치
- 수식어는 피수식어 바로 앞에
- 능동문 우선 (피동 최소화)
- 홑문장 지향 (겹문장은 의도적일 때만)
- 조사·어미 정교하게 (은/는 vs 이/가 구분, -며 vs -고 구분)
- 불필요한 '의', '것', '들', '적' 제거
- 문장 길이 변주 (긴-짧-긴 리듬)
- 한 문단 = 한 생각

## 4. 옵션별 동작

### 기본 (옵션 없음)
1. outline 생성 → 사용자 승인 요청
2. 승인 후 draft 작성 (2-Pass 자동 적용)

### --outline
주제를 받아 구조 설계:
- 전체 흐름 (장르에 맞는 구조)
- 섹션별 핵심 메시지 1줄
- 예상 분량
사용자 승인 후 --draft로 이어갈 수 있음

### --draft
outline 없이 바로 초안 작성. 2-Pass 자동 적용:
- Pass 1 (구조): 흐름, 논거 배치, 전환 자연스러움 확인
- Pass 2 (문체): anti-AI 패턴 제거, 장르 톤 교정, 어미 변주

### --polish
사용자가 붙여넣은 텍스트를 교정:
- 원문의 의미·구조는 유지
- anti-AI 룰셋 기반 문체만 교정
- 주요 변경점을 bullet으로 정리해서 함께 출력

### --output (출력 형식)

출력 형식은 대화 단계에서 자연스럽게 파악한다. 명시적 옵션:
- `--output md` — 마크다운 (기본값)
- `--output brunch` — 브런치스토리 최적화 (문단 줄간격, 헤더 스타일 조정)
- `--output word` — pandoc으로 .docx 변환 (`pandoc output.md -o output.docx`)
- `--output pdf` — Typst 템플릿으로 PDF 생성

**PDF 템플릿 선택** (장르별 기본값 자동 추천, 변경 가능):

| 템플릿 | 스타일 | 기본 장르 |
|---|---|---|
| A — Academic | 2단 컬럼, IEEE 스타일 | 학술 |
| B — Modern | 컬러 헤딩, 깔끔한 리포트 | 기획서·보고서 |
| C — Minimal | 넓은 여백, 세리프, 절제된 타이포 | 에세이·소설 |
| D — Tech Dark | 다크 헤더, 코드 친화적 | 기술문서 |

템플릿 경로: `~/projects/agent-orchestration/templates/typst/template_{A~D}_*.typ`

PDF 생성 명령:
```bash
typst compile output.typ output.pdf
```

### --series {시리즈명}
시리즈 연속 글쓰기 모드:
1. `mcp__obsidian-vault__read_note` → `10-knowledge/writing/series/{시리즈명}.md`
   - 파일 없으면: 템플릿(`series/_template.md`) 기반으로 새로 생성
   - 파일 있으면: 기존 컨텍스트(톤, 키워드, 인물, 편별 기록) 로드
2. 로드된 컨텍스트를 장르·문체 가이드와 함께 적용
3. 작성 완료 후 시리즈 컨텍스트 자동 업데이트:
   - 편별 기록 테이블에 새 행 추가
   - 새로 등장한 인물·키워드 반영
   - "다음 편 메모" 갱신
4. 자기검증 6-4(시리즈 일관성) 자동 활성화

### --vault
완성된 글을 vault에 저장:
- `mcp__obsidian-vault__write_note`
- 경로: `30-projects/writing/{장르}/{날짜}-{슬러그}.md`
- frontmatter: type, genre, topic, date, status(draft/final)
- `--series` 사용 시 series 필드 추가

### --brief
과정 설명 생략. 결과물만 출력.

## 6. 자기검증

초안 완성 후 아래 절차를 반드시 실행:

### 6-1. 분량 실측 (추정 금지)
초안 작성 후 **반드시** 아래 bash 명령으로 실제 글자 수를 측정한다:
```bash
echo '본문 텍스트' | wc -m
```
- 제목, 구분선, 공백 행(`&nbsp;`) 제외한 순수 본문만 측정
- 측정값이 목표 분량의 ±20% 밖이면:
  - 미달: 부족한 장면·단락을 보완하여 다시 작성 후 재측정
  - 초과: 불필요한 부분을 줄인 뒤 재측정

  - brunch: ±15% 범위 적용
  - newsletter: ±15% 범위 적용

### 6-3. 맞춤법 검사 (Pass 3)
본문을 아래 빈출 오류 패턴으로 검사하고, 발견 시 **자동 수정** 후 재출력한다.

**검사 & 자동 수정 규칙:**

| 패턴 | 오류 예 | 수정 | grep |
|---|---|---|---|
| 되/돼 혼동 | 안되는, 되요, 되서 | 안 되는, 돼요, 돼서 | `되요\|되서\|안되` |
| 않/안 혼동 | 하지안, 않먹 | 하지 않, 안 먹 | `하지안\|먹지안\|않먹\|않가` |
| 띄어쓰기 | 할수있, 안되는 | 할 수 있, 안 되는 | `할수있\|할수없\|안되\|못되\|그럼에도불구\|때문이아니라` |
| 이중피동 | 만들어지다, 보여지다 | 만들다, 보이다 | `만들어지\|보여지\|써지\|읽혀지` |

**실행 절차:**
1. 본문에서 위 패턴을 grep으로 검출
2. 발견된 각 항목을 아래 sed 치환으로 자동 수정:
```bash
echo "$TEXT" | sed -E \
  -e 's/안되([는면])/안 되\1/g' \
  -e 's/못되([는면])/못 되\1/g' \
  -e 's/되요/돼요/g' \
  -e 's/되서/돼서/g' \
  -e 's/할수있/할 수 있/g' \
  -e 's/할수없/할 수 없/g' \
  -e 's/보여지/보이/g' \
  -e 's/만들어지/만들/g' \
  -e 's/읽혀지/읽히/g'
```
3. 수정 내역을 bullet으로 표기:
```
맞춤법 수정:
- "안되는" → "안 되는" (띄어쓰기)
- "되요" → "돼요" (되/돼 혼동)
```
4. 수정 0건이면 "맞춤법 검사 통과"로 표기

### 6-5. 문체 검증 (--verify 옵션 사용 시만)

`--verify`가 지정된 경우 아래 두 단계를 추가 실행한다.

**A. 정량 비교**
본문을 임시 파일에 저장 후 bash로 아래 지표를 측정:
```bash
# 문장 수
echo "$TEXT" | sed 's/[.?!]/\n/g' | grep -c .
# 평균 문장 길이 (글자/문장)
echo "$TEXT" | wc -m  # 총 글자 수를 문장 수로 나눔
# 질문문 개수
echo "$TEXT" | grep -ocE '\?|는가\.|인가\.|할까\.|겠는가\.|느냐\.|는지\.|을까\.|ㄹ까\.' | head -1
```

**문체별 지표 사전:**

| 지표 | grep 패턴 | 기대 높은 문체 |
|---|---|---|
| 분석어 | `경제\|비용\|구조\|개념\|논리\|분석\|비율\|산술\|고정비\|변동비\|기회비용\|외부효과\|통계\|수치\|퍼센트\|데이터\|수도권\|집중\|정책\|제도` | 유시민 |
| 감각어 | `빛\|냄새\|온도\|바람\|소리\|눅눅\|진동\|차갑\|뜨겁\|촉각\|피부\|살갗\|감각\|후각\|시각\|떨림\|눈꺼풀\|목덜미\|발바닥\|손등\|체온` | 한강, luma |
| 비유·의인법 | `같은\|처럼\|듯\|만한\|마치\|삼키\|뱉\|닳\|태우\|녹\|무너지\|찢\|갈라지\|흔들\|잠기` | 이외수 |
| 고유명사(브랜드·음악·음식) | `맥주\|파스타\|위스키\|재즈\|앨범\|레코드\|콜트레인\|비틀즈\|샌드위치\|카페\|에스프레소\|문고판\|알덴테` | 하루키 |
| 독자 질문·권유 | `보자\|보겠다\|물어보\|생각해\|아닌가\|아닐까\|할까\|따져보\|환산하면\|의미하는` | 유시민 |
| 열린/모호 결말어 | `모르겠다\|몰랐다\|아직\|그런 것이다\|특별한.*없\|인지도\|글쎄` | 하루키, luma |

측정 결과를 표로 출력:
```
| 지표 | 값 | 기대 문체 일치 여부 |
|---|---|---|
| 총 문장 수 | N | — |
| 평균 문장 길이 | N자 | — |
| 질문문 | N개 | — |
| 분석어 | N개 | 유시민이면 5+, 아니면 0~1 |
| 감각어 | N개 | 한강이면 5+, luma면 2~4, 유시민이면 0 |
| 비유·의인법 | N개 | 이외수면 3+, 아니면 0~1 |
| 고유명사 | N개 | 하루키면 3+, 아니면 0 |
| 독자 질문·권유 | N개 | 유시민이면 2+, 아니면 0 |
| 열린 결말어 | N개 | 하루키/luma면 1+, 유시민이면 0 |
```
기대값과 실측값을 비교해 2개 이상 불일치하면 해당 부분 보완.

**B. 외부 블라인드 검증 (Gemini)**
